## 2026-07-07 — ТЗ-061: возврат средств не отзывал Pro-статус (критический баг)

- **Найдено:** ТЗ-038b обрабатывал `payment.succeeded`/`payment.canceled`, но не
  `refund.succeeded` — отдельное событие ЮKassa. Подтверждено на реальном тесте:
  оплата → Pro → возврат средств → деньги вернулись → Pro остался активным
  бессрочно. Реальная финансовая дыра.
  - По API ЮKassa (доки отдают JS-рендер, структуру собрал через несколько
    источников): объект `refund` содержит `id` (ID возврата, НЕ платежа!),
    `status`, `payment_id` (ссылка на исходный платёж), `amount`. Вебхук —
    та же обёртка `{"type": "notification", "event": "refund.succeeded",
    "object": {...}}`. **Частичные возвраты существуют** (от 1₽).
  - Согласовал с Сашей: отзывать Pro при ЛЮБОМ возврате (полном или частичном),
    без сравнения суммы возврата с суммой платежа — проще и безопаснее для
    бизнеса, чем городить частичное «додержание» доступа.
- **Сделано** (`backend/app/api/v1/payments.py`):
  - `_revoke_pro(user)` — рядом с `_activate_pro`, зеркалит уже существующий
    `POST /admin/users/{id}/revoke-pro` (`tier="free"`, `expires_at=None`).
  - `_verify_and_revoke_refund(refund_id, session)` — та же дисциплина, что
    `_verify_and_sync_payment` (ТЗ-055): обратный `GET /v3/refunds/{id}` к
    API ЮKassa, никогда не доверяем телу вебхука. Из верифицированного ответа
    берём `payment_id`, ищем локальный `Payment`, отзываем Pro у его
    пользователя. Идемпотентно: платёж с уже `status="refunded"` пропускается
    молча — повторная доставка вебхука не долбит лишний раз.
  - `yukassa_webhook` — принимает `refund.succeeded`; важный нюанс:
    `object.id` для событий `refund.*` — это ID **возврата**, а не платежа
    (payment_id лежит внутри объекта возврата) — код различает по типу
    события, какой хелпер вызвать.
  - `Payment.status` — в комментарий добавлено значение `refunded` (миграция
    не нужна — поле `str`, схема через `create_all`, не настоящий DB-enum).
- **Проверено:** новый класс `TestYukassaRefund` (6 тестов) — полный цикл
  оплата→Pro→возврат→Pro отозван; частичный возврат тоже отзывает (прямая
  проверка согласованного решения); идемпотентность повторной доставки;

  тело вебхука врёт о статусе (переверификация показывает `pending`) → Pro

  **145 passed, 2 skipped** (было 139, +6 новых, регрессий нет).

- **Ручные шаги для Саши после деплоя** (вне моего доступа — личный кабинет
  ЮKassa и продовая БД):
  1. **Проверить/включить подписку на `refund.succeeded`** в личном кабинете
     ЮKassa (Интеграция → HTTP-уведомления) — та же страница, где настраивался
