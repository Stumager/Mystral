## 2026-07-12 — ТЗ-065 доп.: живой прогон на docker-compose нашёл 2 реальных бага

- Саша поднял Docker Desktop — прогнал полный цикл через реальные
  Postgres/Redis/backend (не SQLite из тестов). Живые находки:
  1. **`DELETE /admin/users/{id}` падал 500** на любом пользователе с
     платежом — ручной cascade-список в `delete_user` (`backend/app/api/v1/admin.py`)
     никогда не включал `payments` (пробел с ТЗ-038b/055, просто не был
     покрыт тестами) и тем более не знал про новый `refund_requests`.
     Postgres реально проверяет FK (`payments_user_id_fkey`), SQLite в
     тестах — нет, поэтому пробел был невидим до живого прогона.
  2. **Тот же cascade-цикл никогда не удалял ни одной строки на SQLite** —
     сырой `text("DELETE ... WHERE user_id = :uid")` с `{"uid": user_id}`
     (питоновский `UUID`) падает на aiosqlite ("type 'UUID' is not
     supported"); `str(user_id)` (с дефисами) не падает, но и не матчится —
     SQLite хранит UUID-колонки как 32-символьный hex БЕЗ дефисов.
     Эмпирически проверил формат хранения и поправил на `user_id.hex` —
     подходит и SQLite (совпадает формат), и Postgres (uuid-тип принимает
     hex без дефисов на входе).
  - Добавлен `TestDeleteUserCascade` (`test_admin_refunds.py`) — раньше
    `delete_user` вообще не имел тестового покрытия. Полный `pytest` —
    **178 passed, 6 skipped**.
  - Отдельно, во время живой проверки заметил: у этого бэкенда кастомный
    global exception handler (`main.py`) отдаёт ошибки как
    `{"error": status, "message": "..."}`, а не FastAPI-дефолтный
    `{"detail": "..."}`. Моя новая строка в `Profile.tsx` читала `d.detail`
    (никогда не сработало бы) — поправил на `d.message || d.detail`. Это
    системный паттерн по всему фронтенду (LoginScreen, NatalChart,
    ForgotPassword, ResetPassword и другие места в самом Profile.tsx читают
    `.detail` тем же образом) — чинил только свою новую строку, остальное
    вне периметра этого тикета.
  - Живьём подтверждён полный цикл модерации: заявка → статус в профиле →
    список в админке → reject с комментарием (+ защита от повторного) →
    короткое замыкание на уже возвращённом платеже (+ защита от повторного
    approve) → каскадное удаление пользователя с платежами и заявками.
  - **Не проверялся** живой успешный `approve` с реальным списанием у
    провайдера — auto-mode классификатор корректно заблокировал попытку

## 2026-07-13 — ТЗ-059: регистрация — generic "Error", валидация имени, баг ввода времени

### Модуль 1 — generic "Error" вместо реальной причины
- **Root cause:** `LoginScreen.tsx` читал `data.detail` из ответа API, но
  глобальный exception handler бэкенда (`main.py`) отдаёт ошибки как
  `{"error": status, "message": "..."}`, а не FastAPI-дефолтный
  `{"detail": "..."}` — `data.detail` всегда `undefined`, поэтому всегда
  срабатывал fallback `"Error"`. Тот же системный паттерн уже находил и чинил
  точечно в `Profile.tsx` (ТЗ-065-доп.) — здесь это было explicit-в-периметре
  тикета, поэтому почищено полностью в `LoginScreen.tsx` (оба пути: обычная
  регистрация/логин и Telegram-виджет).
  - Разовый обрыв 400 без деталей (о котором писал Саша) — не воспроизвёл и
    не гадал вслепую (как и просили): `_check_ip_rate` отдаёт 429, не 400, так
    что это не rate-limit; сам факт, что реальный текст теперь виден, значит
    при повторном возникновении причина будет сразу читаема, а не потеряна.
  - `res.json()` обёрнут в `.catch(() => ({}))` — если тело не JSON вообще
    (например, апстрим отдал HTML 502-страницу), fallback теперь на общий
    текст (`login.generic_error`, новый i18n-ключ, 6 языков), а не сырой
    `SyntaxError` от `JSON.parse`.
### Модуль 2 — валидация имени
- Было: только `!name.trim()` и `length > 100`. Новое (frontend `validate.ts`
  + backend `_validate_name` в `auth.py`, одинаковые правила в двух местах):
  не пусто после trim, 2-50 символов, минимум одна буква (frontend:
  `/\p{L}/u`; backend: `str.isalpha()` — стандартный эквивалент без
  сторонней `regex`-библиотеки, работает для любого из 6 языков платформы,
  не хардкод конкретных алфавитов), запрет `<`/`>` (XSS в шер-карточках/
  письмах/профиле). **Блеклист "SQL-символов" сознательно не делал** —
  проверил, что `display_name`/имя нигде не конкатенируется в сырой SQL
  (только через ORM, параметризовано) — блеклист просто отсекал бы легитимные
  имена без реальной пользы.
  - Фронтенд: чек-лист под полем имени в реальном времени, по образцу уже
    существующего чек-листа пароля на этой же форме (2 критерия: длина,
    наличие буквы), плюс кнопка submit заблокирована при невалидном имени —
    аналогично паролю.
  - Имя триммится перед сохранением в БД (`req.name.strip()`).
### Модуль 3 — не вводилось время рождения (диагностика → фикс)
- **Диагностика:** поле в `OnboardingModal.tsx` (шаг 1 регистрационного
  флоу онбординга, не сама форма регистрации — время рождения там не
  запрашивается вообще). Реальный `<input type="time">`, но **скрытый**
  (`opacity: 0, pointerEvents: "none"`) и открывался программно через
  `timeRef.current?.showPicker?.()` по клику на отдельную кнопку.
  **Root cause:** `showPicker()` поддерживается не везде (Safari < 16.4,
  вероятно встроенный WebView Telegram на более старых iOS) — при отсутствии
  метода `?.()` молча ничего не делает, без ошибок в консоли. День/месяц/год
  рядом используют другой, надёжный паттерн — реальный `<select>` поверх
  кастомного вида, кликабельный напрямую, без зависимости от JS API. Поле
  времени было единственным местом с `showPicker()`-паттерном во всём фронтенде.
- **Фикс:** убрал скрытый инпут + `showPicker()` целиком, поле времени
  теперь использует ТОТ ЖЕ overlay-паттерн, что день/месяц/год — реальный
  `<input type="time">` поверх стилизованного блока, клик работает через
  штатное поведение браузера с любым native time-контролом, не через
  скриптовый API. Заодно сделал инпут `controlled` (`value=...`) и добавил
  очистку `hour`/`minute` при очистке поля (была невозможна раньше).
### Тесты и проверка
- Backend: 4 новых теста в `test_auth.py` (пустое/только пробелы/1 символ,
  спецсимволы-без-буквы/51+ символов/HTML-тег — оба набора разбиты на 2 теста

- Живая проверка на docker-compose (реальный Postgres/Redis/backend):

### Problems found
- `warm_seo_cache()` (тот же файл) после 3 подряд ошибок делает `return` —
  бросает прогрев **всех** оставшихся страниц во всех типах разом, а не
  только текущего типа. Комментарий из коммита `52ff482` («No point burning
  through error responses when Groq daily limit is hit... resumes next
  server restart») явно предполагал, что непрогретые страницы спокойно
  переживут on-demand генерацию до следующего рестарта — но это предположение
  было неверным именно из-за бага выше, отчего 21 страница вслед за первым
  сбоем оказались 500, а не просто «медленнее генерируются». Не трогал —
  сам по себе этот `return` не 500-ит (это вопрос охвата прогрева, не
  корректности), а фикс try/except выше уже полностью закрывает
  зарепорченный симптом. Флагирую отдельно, если Саша захочет сделать прогрев
  устойчивее к точечным сбоям.
